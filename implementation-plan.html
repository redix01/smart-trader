1. Current Template Context

What the dummy dashboard already gives us

  • Premium dark trading/investment UI with sidebar, sticky header, and animated page transitions.
  • Client-side page switching via local React state instead of URL routing.

2. Laravel + Inertia Target

Bootstrap Decision

  • If this folder is the final app location, first create a Laravel shell here or in a sibling folder.
  • Fastest route is Laravel with Breeze React/Inertia auth scaffolding, then replace generated dashboard UI.
  • Do not port the current SPA routing as-is; each page should become an Inertia route.

Frontend Direction

  • Move reusable layout pieces into resources/js/Layouts and resources/js/Components.
  • Convert page state navigation to real links using Inertia Link and named Laravel routes.
  • Keep the visual language, but replace mock data with page props from controllers/services.

Backend Direction

  • Each dashboard section needs a controller, request validation, service layer, and policy checks.
  • Use shared Inertia props for auth user, verification status, balances snapshot, and nav metadata.
  • Treat trading, mining, staking, and expert-copying as internal products with clear domain models.

6. Admin Management

Authorization model

  • The admin area must not share the same authorization boundary as the user dashboard.
  • Users should only access their own dashboard, balances, requests, subscriptions, and profile actions.
  • Admins should be protected by separate middleware, roles, and permissions, not by normal user checks alone.
  • Admins can access both the admin area and the user-facing dashboard without the normal end-user restrictions.

Routing and layout split

  • Create a dedicated admin route group such as /admin/* with its own navigation and layout.
  • Keep the customer sidebar clean; do not mix admin navigation into the user dashboard shell.
  • Use separate controllers under app/Http/Controllers/Admin and separate Inertia pages under resources/js/Pages/Admin.
  • If admin needs to inspect user views, make that explicit with impersonation or audited view-as-user flows.

Required admin pages

  • Admin dashboard with pending approvals, alerts, totals, and operational queues.
  • User management index and user detail management page.
  • CRUD screens for trades, swaps, deposits, withdrawals, plans, experts, and projects.
  • Settings, permission, and audit log pages for sensitive actions.

Backend requirements

  • Separate admin middleware or role gates for all admin routes.
  • Dedicated policies for create, update, approve, reject, suspend, and configure actions.
  • Audit logging for sensitive admin actions and unrestricted access behavior.
  • Optional impersonation support only if it is logged and intentionally granted.

Implementation rule

  • Build admin as a first-class application area, not as hidden buttons inside the user dashboard.
  • Authorization separation comes before CRUD expansion.
  • Every admin write path should have validation, authorization, and audit coverage.
  • Admin access to unrestricted user modules should be role-based and reviewable.

10. TradingView Widget Integration

Replaced all fake/static chart placeholders with live TradingView embedded widgets across the Dashboard and Trades pages. Both widgets are injected at runtime via useEffect + useRef — no npm package required, no SSR conflicts. The Ticker Tape is shared across both pages; the Advanced Chart adapts its symbol based on the selected pair and market type.

Implementation pattern (both pages)

  • Container div with a ref holds the widget. On mount (or symbol change), the container is cleared and a new <script> is appended with the JSON config embedded as its innerHTML.
  • TradingView's embed scripts self-render an <iframe> into their parent element.
  • Cleanup function in useEffect clears innerHTML on unmount to prevent duplicate widgets.
  • Chart dependency array is [tvSymbol] — reloads automatically on pair/market change. Ticker Tape dependency is [] — mounts once.
